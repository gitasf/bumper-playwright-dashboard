The 30-second model

Three pieces, one data flow

Everything hangs off the live ingest path: a test finishes in CI, the reporter posts it, the worker writes it to Postgres + R2 and broadcasts the delta to any browser watching the run.

Playwright CI ──▶ @wrightful/reporter ──▶ Worker /api/runs/* ──▶ Postgres + R2 ──▶ void/ws rooms ──▶ live dashboard

@wrightful/reporter

The Playwright reporter. Streams results + artifacts as each test reaches its final outcome (retries aggregated into flaky). Published via Changesets.

@wrightful/dashboard

The app (apps/dashboard), built on Void: Hono file-based API in routes/, SSR pages in pages/, Drizzle schema in db/schema.ts. This is where you'll spend your time.

packages/e2e

Playwright E2E — a demo suite for dogfooding and the canonical dashboard suite that self-boots a real Void dashboard.

Get running

Local setup

The dashboard is Postgres-only. The happy path boots a local Postgres via Docker, writes .env.local, migrates the DB, starts the dev server, and seeds a demo team / project / API key + monitors.

happy path — needs Docker
$ pnpm install
$ pnpm setup:local        # Docker Postgres + .env.local + migrate + seed
$ pnpm dev                # dashboard on http://localhost:5173

Optional — seed months of synthetic history instead of the small fixture set:

more data
$ pnpm setup:local --history                    # 3 months
$ pnpm setup:local --history --history-months 6  # 6 months
▸ No Docker? Native Postgres path alternative setup

setup:local hard-requires Docker for any localhost DB URL. Without it, run a native Postgres and do the post-Docker steps by hand:

native postgres
# 1. Install + start Postgres 16 (matches CI's postgres:16)
$ brew install postgresql@16 && brew services start postgresql@16

# 2. Create the role + db (same creds as docker-compose.pg.yml)
$ psql -d postgres -c "CREATE ROLE wrightful LOGIN PASSWORD 'wrightful' SUPERUSER;"
$ psql -d postgres -c "CREATE DATABASE wrightful_dev OWNER wrightful;"

# 3. Point .env.local at it, then migrate + seed manually
$ export DATABASE_URL="postgresql://wrightful:wrightful@localhost:5432/wrightful_dev"
$ cd apps/dashboard && npx void db reset        # wipe + migrate
$ cd ../.. && pnpm dev &                           # leave running on :5173
# once /api/auth/get-session responds:
$ cd apps/dashboard
$ WRIGHTFUL_URL=http://localhost:5173 node scripts/seed-demo.mjs
$ WRIGHTFUL_URL=http://localhost:5173 \
    WRIGHTFUL_TOKEN=$(node -e "console.log(require('./.env.seed.json').apiKey)") \
    node scripts/upload-fixtures.mjs
trap

Don't try to dodge the Docker check with [::1] in DATABASE_URL — the void CLI's connection parser can't handle the bracketed IPv6 literal (ENOTFOUND [::1]). Keep localhost in the URL and run the steps manually.

A hosted Postgres (e.g. a free Neon branch) also works — set DATABASE_URL to it and setup:local skips Docker entirely.

The mental model & the gotchas that bite

Tenancy is logical, not physical

will bite · security

There's no per-team boundary. Every query against a run-scoped table (runs, testResults, artifacts, …) must filter by projectId (and teamId where present). The branded AuthorizedProjectId / TenantScope in src/lib/scope.ts make forgetting a type error. Start tenant loaders with requireTenantContext(c); never do raw id lookups.

Postgres dual-dialect trap

will bite · correctness

node-postgres returns int8/numeric as strings; the fast pglite test lane returns numbers. Coercions live in numericSql (src/lib/db/sql-ops.ts) and the raw-read cast(… as integer) idiom. A real postgres:16 CI leg + pg-integration.test.ts exist to catch what green pglite tests hide — don't trust the fast lane alone.

The wire contract is duplicated

keep in mind

TS interfaces in packages/reporter/src/types.ts and Zod schemas in apps/dashboard/src/lib/schemas.ts must stay in sync. The canary is contract.test.ts — it parses reporter-built payloads through the dashboard's schemas.

Two auth systems coexist

keep in mind

Better Auth sessions (void/auth) for the dashboard UI; Bearer API keys for ingest (/api/*). Keys are SHA-256 hashed, looked up by 8-char prefix. Reporter requests also carry X-Wrightful-Version: 3.

Ingest is a deep module

convention

Handlers under routes/api/runs/* are auth + translation only. The real pipeline lives behind openRun / appendRunResults / completeRun in src/lib/ingest.ts; multi-statement writes go through runBatch (a transaction).

UI: reuse before hand-rolling

convention

~50 Base UI wrappers in src/components/ui/ — use them; don't import @base-ui-components/react directly from page code. Tailwind v4 (no config; tokens in src/styles.css under @theme). New components: npx shadcn@latest add @coss/<name>. URL state via useSearchParam — no nuqs.

Worklogs are mandatory

process

Any significant change gets a dated entry in docs/worklog/ (YYYY-MM-DD-short-description.md). This is how the project records decisions beyond git log — future you relies on it.

Don't re-open these

Settled architectural decisions

Read docs/worklog/void-migration-consolidated.md first. The durable calls, not to be re-litigated:

  • One shared Postgres DB with logical tenancy — not per-team Durable Objects.
  • void/ws rooms are the only realtime transport — SSE was removed (see docs/adr/0001-realtime-websocket-rooms.md).
  • The store moved D1 → Postgres (docs/worklog/2026-06-16-postgres-only.md), but "single shared store, logical tenancy" still stands.

Worklogs dated before the Void migration describe the prior rwsdk / Durable-Object / Kysely architecture and are historical — don't take them as current.

Where the project is

Current state

Pre-launch, zero users. The core loop — streaming ingest → analytics → flaky detection → artifacts → realtime — is solid, and roadmap Tiers 1–3 are largely shipped: metering/quotas, retention, GitHub Checks, tag filtering, flaky quarantine, ownership/CODEOWNERS, run-diff, export API, TCP/ping monitors, granular RBAC, audit logs.

Recent work has been consolidation, not features: the D1→Postgres migration, own-account Cloudflare deploy support, and two "architecture deepening" refactor passes. See docs/roadmap/README.md for what's left.

deferred

SSO/SAML — @better-auth/sso isn't Cloudflare-Workers-compatible. Scaffolding was built then reverted; the blocker + how to finish it are documented in docs/roadmap/3.3-sso-saml.md.

Schema/migrations are still malleable (void db generate, destructive OK) — the frozen-migration rule only snaps on at first production deploy.
